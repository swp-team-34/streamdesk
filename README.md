# StreamDesk

StreamDesk - рабочая система для event/production-команд: склад оборудования, задачи, проекты, сметы, схемы подключения, мониторинг машин и администрирование компаний в одном интерфейсе.

Проект рассчитан на работу нескольких компаний в одной платформе. Пользователь может работать лично, создать свою компанию или войти в существующую компанию по инвайт-ссылке. Данные компаний должны быть изолированы: задачи, склад, агенты мониторинга, сметы и схемы не должны попадать в чужое окружение.

## Цель продукта

StreamDesk должен помогать менеджеру, инженеру и владельцу быстро собрать мероприятие: понять ТЗ, оценить оборудование и людей, подготовить смету, превратить ее в схему подключения, выдать задачи команде и проконтролировать состояние компьютеров на площадке.

Главная идея - меньше ручной рутины и больше готовой структуры:

- оборудование на складе с фотографиями, штрихкодами, инвентарниками и техническими характеристиками;
- сметы с историей версий, экспортом и понятными блоками;
- схемы подключения по зонам, портам, входам/выходам и кабелям;
- канбан-доски с плавным перемещением карточек и общими обновлениями для команды;
- мониторинг компьютеров через уникальные агент-файлы компании;
- админ-панель для управления пользователями, компаниями, доступами и приглашениями.

## Техническое задание

### Роли и доступы

- Владелец платформы видит пользователей, компании и может помогать с доступами, блокировать или удалять пользователей.
- Владелец компании управляет сотрудниками, инвайт-ссылками, правами доступа, досками и окружением компании.
- Администратор компании может управлять складом, задачами, проектами и сотрудниками в рамках выданных прав.
- Сотрудник работает только в доступных ему досках, проектах, задачах и складских операциях.
- Личный пользователь может работать в своем окружении и позже вступить в компанию по приглашению.

### Компания и инвайты

- Инвайт-ссылка должна быть доступна владельцу компании всегда.
- Ссылку можно активировать по запросу, например на 24 часа, и отправить сотруднику.
- После регистрации по ссылке пользователь автоматически привязывается к нужной компании.
- У пользователя может быть личное окружение и членство в компании.
- Данные разных компаний не смешиваются даже при одинаковых фамилиях, именах или похожих аккаунтах.

### Task Manager / Kanban

- Можно создавать несколько досок и выбирать, кто из сотрудников их видит.
- В доске можно создавать, переименовывать, удалять и скрывать категории/колонки.
- Карточки задач перетаскиваются мышью в нужную колонку и позицию.
- Перемещение должно быть плавным, без сброса горизонтального скролла, скачков карточек и лишних фоновых обновлений.
- Открытая карточка не должна прыгать при переключении вкладок "Инфо", "Чат", "История" и при добавлении комментариев.
- В карточке задачи должны быть описание, ответственные, аватарки, чекбоксы, подзадачи, файлы, комментарии и история.
- Общие доски должны обновляться у всех участников компании.

### Проекты

- Проект может быть связан с доской.
- Если проект есть в доске, он должен отображаться во вкладке проектов.
- В проектах нужна статистика по задачам, колонкам, статусам, ответственным и срокам.
- Удаление проекта или колонки требует подтверждения.

### Склад оборудования

- Карточка оборудования показывает название, модель, тип, серийный номер, инвентарник, штрихкод, статус, место, владельца/кто забрал, технические характеристики и комментарии.
- Фотографии отображаются аккуратно при раскрытии карточки, если они загружены в компании.
- Инвентарник формируется автоматически из категории и модели, например `cam_013453`, `pc_000124`, `lens_534628`.
- Штрихкод из системы должен находить оборудование на складе.
- Можно выгружать пачки штрихкодов в Excel.
- Можно выгружать технические характеристики: название, инвентарник, характеристики.
- Оборудование, добавленное агентом мониторинга, должно появляться как компьютер с понятными характеристиками CPU, RAM, disk, GPU, IP и hostname.

### Сметы

- Смета должна собираться по ТЗ мероприятия и не ограничиваться только тем, что есть на складе.
- Если оборудования нет, позиция добавляется как аренда/субподряд, чтобы менеджер поставил цену или нашел подрядчика.
- В смете нужны блоки: звук, видео, свет, компьютеры/IT, персонал, логистика/доставка, расходники, прочее.
- Должны учитываться примерное время подготовки, доставки, монтажа, демонтажа, количество машин и расстояние.
- Нужна история смет, чтобы вернуться к старой версии и отредактировать.
- Экспорт должен быть читаемым: блоки, позиции, количество, цена, коэффициенты, комментарии и итог.
- AI-помощник должен помогать понимать ТЗ и подбирать возможные позиции, но не писать в документе внутренние источники ценовых ориентиров.

### Схемы подключения

- Схему можно создать из сметы.
- Оборудование должно раскладываться по зонам на центральной части холста, с запасом пространства между блоками.
- Кабели должны быть читаемыми, не лежать друг на друге и иметь подписи по типу сигнала.
- AI-помощник должен учитывать модель оборудования, входы, выходы, типы портов и примерный путь сигнала.
- Инженер должен получить почти готовую схему и при необходимости только поправить детали.
- Схемы можно сохранять, экспортировать и удалять.

### Мониторинг и агенты

- Вкладка "Серверы" убирается из основного сценария, остается мониторинг.
- Для каждой компании можно скачать уникальные `.bat`-файлы агента.
- Файл агента должен быть привязан к компании, чтобы компьютер не появился в чужом окружении.
- Агент передает состояние компьютера: hostname, IP, CPU, RAM, disk, GPU, процессы, uptime и время последнего обновления.
- При запуске можно включить или не включить автозапуск.
- Графики мониторинга должны быть понятными и стабильными.

### Уведомления

- Текст уведомлений должен быть в корректной кодировке.
- Цвета уведомлений должны быть читаемыми в темной и светлой теме.
- Важные ошибки должны показывать понятную причину и действие.

## Технологический стек

### Frontend

- React 18
- TypeScript
- Vite
- TanStack Query
- Wouter
- Tailwind CSS
- Radix UI
- Lucide React
- React Hook Form
- Recharts
- Konva / React Konva для схем подключения
- jsPDF, PDFKit, docx, html2canvas для экспорта
- html5-qrcode и JsBarcode для штрихкодов

### Backend

- Node.js
- Express
- TypeScript
- WebSocket (`ws`)
- Passport / Passport Local
- Express Session
- Multer для загрузки файлов
- Zod для валидации
- Drizzle ORM

### Database

- PostgreSQL
- Drizzle schema в `shared/schema.ts`
- Миграции и серверные SQL-исправления в `tmp-drizzle/`

### Интеграции

- YouGile - внешняя синхронизация задач, пока локальный канбан остается основным рабочим режимом.
- Hugging Face Inference - AI-помощники для смет и схем.
- vMix / OBS / эфирные сервисы - мониторинг и служебные сценарии.
- SSH/PM2/Nginx - production-деплой.

## Структура проекта

```text
client/                 frontend-приложение
client/src/pages/       страницы StreamDesk
client/src/components/  переиспользуемые компоненты
client/src/lib/         клиентские утилиты
server/                 API, авторизация, интеграции, мониторинг
shared/                 общие схемы и типы
scripts/                деплой, удаленные команды, служебные скрипты
tmp-drizzle/            SQL для настройки/исправления БД
docs/                   дополнительные материалы
```

## Деплой

Production-сборка формируется командой:

```bash
npm run build
```

После загрузки на сервер приложение запускается через PM2. Перед обновлением production нужно сохранить существующую базу данных и не перетирать пользовательские данные.

Минимальный чек-лист деплоя:

- собрать frontend и backend;
- загрузить runtime bundle на сервер;
- применить только безопасные миграции;
- перезапустить PM2-процесс;
- проверить `/api/health`;
- открыть интерфейс и проверить логин, компанию, склад, задачи и сметы.

## Критерии качества

- Пользователь не видит данные чужой компании.
- Регистрация по инвайту сразу приводит пользователя в нужную компанию.
- Канбан не сбрасывает карточки, названия, комментарии и горизонтальный скролл.
- Комментарий в задаче добавляется без визуального "обновления страницы".
- Склад на мобильных и десктопах не имеет наложений элементов.
- Фото оборудования отображаются только там, где помогают карточке, а не ломают сетку.
- Смета читается как коммерческое предложение, а не как сырой список строк.
- Схема подключения понятна инженеру без ручной расстановки всего с нуля.
- Агент мониторинга привязан к компании и не создает чужие компьютеры.
- Ошибки и уведомления читаются на всех темах.

## Замечания по эксплуатации

- В production лучше использовать постоянное хранилище сессий, а не MemoryStore.
- HTTPS обязателен для безопасного входа по паролю.
- AI-ключи и SSH-доступы хранятся только в окружении сервера.
- YouGile-синхронизация не должна удалять локальные задачи при ошибках внешнего API.
- Перед крупными изменениями схемы БД нужно делать резервную копию.
