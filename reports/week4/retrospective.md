# Sprint Retrospective

## What went well

- **Task allocation were allocated correctly.**
- During the sprint perfect task managing was demonstrated by team leader that allowed us to close more user-stories and fix some bugs of product for the current sprint
- **Fast task completion**
- During the sprint each team member finished task earlier than marked deadline, so the process of workflow was more comfortable.
- **Improved team communication**
- Each team member take the role that they give more benefit for the whole team, each of them know their responsibilities and skill set, and that minimized confusion during the sprint.

## What did not go well

- **Challenges with identifying clear acceptance criteria for the PBI**
- For some PBI it is difficult to identify solution for the current problem, because it has several wats how it can be solved.
- **Tracking the progress of the sprint was difficult for the team leader.**  
- There was a lack of transparency: it was difficult to quickly understand what stage each task was at and who was working on what
- **There were synchronization problems when implementing the calendar and task manager.**  
- The calendar should be linked to the task manager, because the calendar should display task descriptions in relation to dates. However, at the same time, the task manager was being developed in parallel - new functionality was added to it and bugs were fixed. Because of this, the calendar and task manager developed independently, which led to difficulties with their subsequent integration.

## Changes implemented from the previous Retrospective

- **Improvements from Sprint 1**
- Our team have several problems with communication and task allocation for the previous sprint, but in this sprint the team leader complete deep analysis and allocated task in the most proper way, so the process of workflow become more comfortable

## Concrete process improvements for the next Sprint

- **Agree on one solution before taking a task into a sprint.**
- If we see several ways, we quickly discuss and choose one. And we immediately write clear acceptance criteria, without vague formulations.
- **Update task status daily.**
- If two parts depend on each other, agree on the connection before starting work, so we don’t build things that don’t fit together.**
